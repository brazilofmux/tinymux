# Profile Analysis — 2026-03-12 (brazil branch)

## Setup

- **Build**: `-pg` profiling enabled (CFLAGS/CXXFLAGS/LDFLAGS)
- **Tool**: `sprof` (LD_PROFILE) — profiles shared libraries via ld.so
- **Workload**: Full smoke suite — 593 test cases (538 functional + 55 benchmarks × 10K iterations)
- **Architecture**: libmux.so (core) + engine.so (eval/commands) + netmux (driver)
- **Branch**: brazil (post-mux_string deletion, co_* Ragel functions, AST evaluator)

### Key Finding

**99.94% of engine.so time is SQLite overhead.** The top 6 functions are all
SQLite schema initialization/validation. The actual game engine — AST eval,
function dispatch, commands, JIT — registers at 0.06% combined.

`ast_eval_subst` (the AST evaluator): 0.02s / 3,061,832 calls = unmeasurably fast.
`do_reference` (permission checks): 0.15s / 43,440,493 calls = also trivial per-call.

### SQLite Schema Overhead Breakdown

The call chain is:
```
sqlite3_exec() → sqlite3Init() → sqlite3InitOne() → sqlite3StartTable() / sqlite3AnalysisLoad()
```

Every `sqlite3_exec()` call validates the schema. With 17 tables (objects,
attributes, attrnames, metadata, channels, channel_users, player_channels,
mail_headers, mail_bodies, mail_aliases, code_cache, plus indexes), each
schema validation parses all CREATE TABLE/INDEX statements.

**Write-through s_*() accessors use prepared statements** (good — schema is
cached). But several paths still use `sqlite3_exec()` for:
- `BEGIN` / `COMMIT` / `ROLLBACK` (transaction control)
- `DELETE FROM objects/attributes/attrnames` (bulk clear)
- `PRAGMA optimize` / `PRAGMA wal_checkpoint`
- Mail module operations (read, list, stats, review)

### Recommendation: Prepared Statements for Transactions

Replace the three most-called `sqlite3_exec()` patterns with prepared statements:

```cpp
// Current (triggers schema validation each time):
sqlite3_exec(m_db, "BEGIN", nullptr, nullptr, nullptr);
sqlite3_exec(m_db, "COMMIT", nullptr, nullptr, nullptr);

// Proposed (schema cached after first call):
sqlite3_stmt *m_stmtBegin, *m_stmtCommit, *m_stmtRollback;
// Prepare once at Open(), finalize at Close()
```

The mail module should also convert its `sqlite3_exec()` calls to prepared
statements, especially for operations called per-message (read, list, stats).

### Key Finding

libmux.so's profile is dominated by **COM/module infrastructure** — the
stubslave inter-process marshaling framework. No stubslave process is
actually running during smoke tests, so these are initialization/registration
costs distributed across many small calls.

**Notably absent**: `mux_exec`, `ast_eval`, `co_*` functions, `safe_str`,
`alloc_lbuf`, parse cache — the actual eval hot path. These are either too
fast to sample, or live in engine.so (which has its own profile).

`RandomINT32` at 1.40% is the only eval-related function visible — it's
called heavily by the benchmarks (10K iterations × many `rand()` calls).

## What This Means

1. **The eval path is not the bottleneck.** AST evaluation, function dispatch,
   co_* Ragel functions, and the JIT are all negligible in wall-clock time.
   The "slowness" perception is likely SQLite schema overhead during startup
   and during write-through operations.

2. **SQLite schema caching is the #1 optimization target.** Converting
   `sqlite3_exec()` calls to prepared statements would eliminate repeated
   schema parsing. This is a mechanical change with massive impact.

3. **The mail module is the second-largest consumer** after SQLite schema
   init. Mail operations (read, list, stats, review, reply) together account
   for ~0.05% of engine.so time — small in absolute terms but large relative
   to everything else outside SQLite.

4. **COM infrastructure in libmux.so is exercised but idle.** The module
   registration/marshaling code runs during startup but does no real work
   without a stubslave process. This is expected overhead.

5. **This finding likely affects 2.13 as well** — the SQLite backend and
   `sqlite3_exec()` patterns are similar. Profiling the 2.13 release branch
   before BETA would confirm.
