### CORRECTION: Histogram Sampling Artifact

**The initial reading was misleading.** The 99.94% attributed to SQLite is a
**code size bias** in sprof's histogram-based sampling, NOT actual execution time.

Evidence:

| Function | Code size (bytes) | Profiler % |
|----------|-------------------|------------|
| `sqlite3StartTable` | 2,146 | 36.96% |
| `sqlite3InitOne` | 1,931 | 39.52% |
| `sqlite3_exec` | 1,576 | 10.75% |
| `sqlite3AnalysisLoad` | 442 | 9.19% |
| `sqlite3Init` | 211 | 2.53% |
| **Total** | **6,306** (0.25% of .text) | **99.94%** |

The SQLite amalgamation compiles as one translation unit with large `static`
functions. sprof's histogram bins cover instruction ranges â€” large functions
get more bins and thus more "time" regardless of actual execution frequency.
The `calls = 0` column confirms sprof couldn't count actual invocations
(static symbols are invisible to the call-counting mechanism).

The 400x over-attribution (0.25% code â†’ 99.94% "time") is a classic sprof
pathology with amalgamation builds.

### What About the Write-Through Path?

All hot-path writes use **prepared statements** (sqlite3_step/sqlite3_reset):
- `UpdateSingleField()` for s_Location, s_Contents, s_Owner, etc.
- `PutAttribute()` for atr_add_raw write-through
- `DelAttribute()` for attribute deletion

`sqlite3_exec()` is used only for:
- Startup pragmas (7 calls, once at Open)
- Schema creation (once at Open)
- `BEGIN`/`COMMIT`/`ROLLBACK` (batch operations only, not per-write)
- Bulk `DELETE FROM` (rare admin operations)

**The write-through path is already optimal.** No per-write schema validation.

## Conclusions

1. **sprof histogram sampling is unreliable for amalgamation builds.**
   The SQLite amalgamation's large static functions create a code-size bias
   that dominates the flat profile. Call-count data is more trustworthy.

2. **The eval path is fast.** `ast_eval_subst` at 3M calls / 0.02s and
   `do_reference` at 43M calls / 0.15s show nanosecond-scale per-call costs.

3. **The write-through path is already optimal.** All hot-path SQLite
   operations use prepared statements. No repeated schema validation.

4. **We are NOT slower due to recent changes.** The co_* replacements and
   mux_string deletion did not introduce measurable overhead. The "slowness"
   perception is the 55 benchmarks Ã— 10K iterations adding ~2 minutes of
   CPU-bound work to the smoke run.

5. **Need better profiling tools for meaningful analysis.** Options:
   - Lower `perf_event_paranoid` to enable `perf record` (best option)
   - Build a static binary (no .so) for accurate gprof
   - Use `-finstrument-functions` with custom counters
   - Add targeted `clock_gettime()` instrumentation to specific paths

## Profiling Limitations

- **sprof histogram bias**: Large functions get disproportionate "time"
  based on code size, not execution frequency. Amalgamation builds
  (sqlite3.c) are especially affected.
- **Static symbols invisible to call counter**: sprof shows `calls = 0`
  for static functions, meaning actual invocation counts aren't available.
- **perf_event_paranoid = 4**: Linux perf is locked out without root,
  preventing the most accurate profiling tool from being used.
- **gprof + shared libraries**: gprof only profiles the main binary;
  engine.so and libmux.so (where all hot code lives) are invisible.
